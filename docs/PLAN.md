# 「発電王」サポートアプリ 開発計画書

最終更新: 2026-07-05

## 0. このドキュメントの位置づけ

『エネルギーミックス・ミッション』ルールに沿ってカードゲーム「発電王」を授業（特に参観日）で実施する際に、
- ミッションカード／イベントカードの抽選・提示
- 各グループが場に出したカードの自動判定（クリア可否・順位付け）

を担当する補助アプリを作るための計画書。実装に入る前に、範囲・データ構造・画面・進行フロー・当日の運用まで合意しておき、後戻りを減らすことを目的とする。

---

## 1. 目的

- 授業内カードゲーム「発電王」の**審判役**をアプリに担わせる。
- 教員が計算ミスやルール適用漏れを気にせず進行に集中できるようにする。
- 参観日で保護者にも「今、何が起きているか」が一目でわかる大画面表示を提供する。

## 2. スコープ（やること／やらないこと）

依頼文から、アプリが担当すべき範囲は明確に絞れる。

### アプリが担当する（IN）
| # | 機能 |
|---|---|
| 1 | ミッションカード（社会の要求）のデッキ管理・ランダム抽選・表示 |
| 2 | イベントカード（トラブル）のデッキ管理・ランダム抽選・表示 |
| 3 | 発電所カードのマスタデータ管理（ステータス・特殊効果） |
| 4 | 各グループが場に出したカードの入力 |
| 5 | イベント効果／カード特殊効果を適用した上での自動判定（クリア可否・順位） |
| 6 | ラウンドごとの結果表示・スコアボード |

### 物理カード側で行う（OUT・アプリは関与しない）
- 20枚の共有山札からのドラフト（配札）
- 各プレイヤーの手札管理そのもの（何を持っているかはプレイヤー自身が把握）
- カードの印刷・実物の準備

この切り分けにより、アプリは「ゲームエンジン」ではなく「審判・電卓・演出」に専念すればよく、実装範囲が現実的になる。

## 3. カード情報の構造化

添付の4枚（LNG火力／原子力／洋上風力(浮体式)／石油火力）から、発電所カードは共通フォーマットを持つことが分かる。

### 3.1 発電所カード
```
PlantCard {
  id: string
  name: string                 // 例: "LNG火力発電所"
  category: string             // 例: "系統安定" | "次世代GX" | "再生可能" | "石油燃料"
  catchCopy: string            // タイトル下の一言
  description: string          // 発電方式の説明文
  stats: {                     // レーダーチャートの5軸。各0〜5
    output: number             // 発電量
    safety: number             // 安全性
    selfSufficiency: number    // 自給率
    efficiency: number         // 効率性
    environment: number        // 環境
  }
  effect: {
    keyword: string            // 【バックアップ】【ベースロード】【お天気依存】など
    text: string                // 表示用の効果文
    rule: EffectRule            // プログラムが解釈する構造化ルール（4.2参照）
  }
}
```

### 3.2 ミッションカード
```
MissionCard {
  id: string
  title: string                 // 例: "国際環境サミット"
  flavorText: string
  condition: {
    output: { min: number }                              // 目標発電量
    secondary?: { stat: "environment"|"selfSufficiency"|"efficiency"|"safety", min: number }
  }
}
```
例:
- 国際環境サミット: `output.min = 12`, `secondary = { stat: "environment", min: 15 }`
- 大寒波襲来: `output.min = 25`, `secondary = { stat: "selfSufficiency", min: 8 }`

### 3.3 イベントカード
```
EventCard {
  id: string
  title: string                 // 例: "長雨（曇天）"
  text: string
  rule: EffectRule               // 盤面全体に適用される効果
}
```

## 4. 効果（特殊効果／イベント効果）の共通表現

4枚のサンプルだけでも効果の種類はバラバラ（「他プレイヤーの状態を参照する」「天候依存」「使うかどうか選べる」）。
自由記述のテキストのままでは自動判定できないため、**効果を「条件＋対象＋補正」の組で構造化**し、あとから20枚ぶん追加してもロジックを直さずに済むようにする。

```
EffectRule {
  trigger:
    | "always"                              // 原子力: 常時
    | "eventIs(<eventId>)"                  // 洋上風力: 特定イベント発生時
    | "anyPlayerRenewableStopped"           // LNG: 誰かの再エネが停止した時
    | "ownFieldHasStoppedCard OR season=='冬'"  // 石油: 自分の場に停止カードがある/冬
    | "categoryIs(<category>) AND eventIs(<eventId>)"  // イベント側: 太陽光×長雨 など
    | "statLte(selfSufficiency, 2)"         // イベント側: 自給率2以下のカードごとに

  optional: boolean        // true = プレイヤーが使うかどうか選べる(【バックアップ】系)
  target: "self" | "matchingCards" | "allFieldCards"
  modifier: { stat: keyof Stats, op: "add" | "set", amount: number | { ifEventIs: Record<eventId, number> } }
}
```

これにより、たとえば：
- LNG【バックアップ】→ `trigger: anyPlayerRenewableStopped, optional: true, modifier: { stat: output, op: add, amount: 4 }`
- 原子力【ベースロード】→ `trigger: always, modifier: { immuneToWeatherEvents: true }`
- 洋上風力【お天気依存】→ `trigger: eventIs('windless'), modifier: { stat: output, op: set, amount: 0 }`
- 石油【バックアップ】→ `trigger: ownFieldHasStoppedCard_or_winter, optional: true, modifier: { stat: output, op: add, amount: { default: 5, ifEventIs: { fuelPriceSurge: 3 } } }`
- イベント「燃料価格の高騰」→ `trigger: statLte(selfSufficiency, 2), target: matchingCards, modifier: { stat: efficiency, op: add, amount: -2 }`

と、すべて同じ枠組みで表現できる。**「20枚すべての効果テキストを、この形式に落とし込む」作業が最初のヤマ場**になる。

## 5. 判定ロジック（計算パイプライン）

1グループぶんの判定は次の順序で計算する。

1. 対象グループが場に出したカード（最大3枚、または累積分）を取得
2. 各カードの `stats` をコピー（作業用）
3. **イベント効果を適用**（例: 太陽光カードの発電量を一律0にする／自給率2以下のカードの効率性を-2）
4. **カード自身の特殊効果を適用**
   - `optional: false` の効果（【ベースロード】など）は自動適用
   - `optional: true` の効果（【バックアップ】など）は条件を満たしていればUI上でON/OFFトグルを出し、プレイヤーの選択に委ねる
5. 全カードの `stats` をグループごとに合計
6. ミッション条件と比較
   - `output合計 >= condition.output.min`
   - `condition.secondary` があれば `該当stat合計 >= secondary.min`
   - 両方満たせば「クリア」
7. クリアしたグループが複数いる場合の順位付け（タイブレーク）
   - **【確定】効率性合計→安全性合計の順で比較する**（同点ならさらに次点の指標で比較）
8. 判定結果（クリア/未達成、各stat内訳、順位）を画面に表示

内訳は「素の合計 → イベント補正 → 特殊効果補正 → 最終合計」を全部見せる設計にし、生徒・保護者が「なぜこの結果になったか」を追えるようにする。

なお「場に出せるカードは最大3枚まで」は、**【確定】ミッションが変わる（ラウンドが進む）たびに全グループの場をリセットし、新たに最大3枚を出し直す**方式とする。これにより判定は常に「そのラウンドで出した最大3枚」だけを対象にすればよく、ロジック・UIともにシンプルになる。

## 6. 全体アーキテクチャ（運用形態：各グループがタブレットで入力）

運用形態は「**各グループが自分のタブレットで場のカードを入力し、教員機（進行役）が全体を管理・大画面表示する**」形に決定した。単一デバイスの完結アプリではなく、**進行役1台＋グループ台数ぶんの端末がリアルタイムに同期するクライアント/サーバー構成**が必要になる。

```
                会場のローカルネットワーク（教員PCのモバイルルーター／学校Wi-Fi等）
                              │
        ┌─────────────────────┼─────────────────────┐
        │                     │                     │
  ホスト画面（教員PC）   グループ1タブレット   グループ2タブレット …（3〜4台）
  ・大画面/プロジェクター   ・ブラウザでURLを開くだけ（アプリのインストール不要）
        │                     │                     │
        └─────────── サーバー（教員PC上で起動、Node.js） ───────┘
                 ゲーム状態（現在のミッション/イベント、各グループの提出）を一元管理し
                 WebSocketでリアルタイムに全端末へ配信する
```

- サーバーは**教員のノートPC上で当日その場で起動**し、タブレットは同じローカルネットワーク上でブラウザからアクセスするだけにする（アプリインストール不要、学校の端末制限にも影響されにくい）。インターネット接続には依存させず、教員PC＋Wi-Fiルーター（モバイルルーター等）だけで完結させる。
- 進行の流れ：①教員がホスト画面で「ミッションをめくる」「イベントをめくる」→ 全タブレットに自動反映 ②各グループはタブレットでカードを選んで「提出」 ③教員画面で全グループの提出状況（提出済み/未提出）を確認 ④「判定」ボタンで一括判定し、結果を全端末＋ホスト画面に表示。

## 7. 画面構成（案）

### ホスト画面（教員PC・大画面投影用）
| 画面 | 内容 |
|---|---|
| ① タイトル画面 | ロゴ、参加グループ数・ラウンド数設定、ゲーム開始（同時にサーバー起動） |
| ② ミッション/イベント公開画面 | 「ミッションをめくる」→条件表示、「イベントをめくる」→効果表示（演出込み） |
| ③ 提出状況モニター | 各グループの提出済み/未提出、判定実行ボタン |
| ④ 判定結果画面 | 全グループの内訳（素点→イベント→特殊効果→合計）、クリア可否、順位 |
| ⑤ スコアボード | 複数ラウンドを通した累積成績・最終順位 |
| ⑥ カード図鑑（管理者用） | 20枚のステータス・効果を一覧確認・修正 |

### グループ画面（各タブレット）
| 画面 | 内容 |
|---|---|
| a. 参加画面 | グループ名/番号を選んで入室 |
| b. 現在のミッション/イベント表示 | ホストと同じ内容をタブレットでも確認できる |
| c. 場入力画面 | 20種のカード一覧から選択（最大3枚）、選択中カードのステータスプレビュー、任意効果（【バックアップ】等）のON/OFFトグル、「提出」ボタン |
| d. 待機画面 | 提出後、他グループの提出待ち／判定待ちを表示 |
| e. 結果画面 | 自グループの内訳・クリア可否・順位を表示 |

大画面（プロジェクター）投影を前提にホスト画面は文字サイズ・コントラストを強めにする。タブレット側はグループで1台を囲んで操作する想定のため、ボタンを大きめにする。

## 8. 技術スタック（提案）

- **フロントエンド**: React + TypeScript + Vite。ホスト画面／グループ画面は同じコードベースの別ルートとして実装（例: `/host`, `/group/:groupId`）
- **バックエンド**: Node.js + Express + Socket.io（または同等のWebSocketライブラリ）。ゲーム状態（現在のミッション/イベント/ラウンド数、各グループの提出内容・判定結果）をサーバー側で一元管理し、状態変化をWebSocketで全クライアントにブロードキャストする
- **判定ロジック**：4章のEffectRuleエンジンはサーバー側に置き、全クライアントが同じ結果を見られるようにする（クライアント側での計算はプレビュー用途に限定）
- カード／ミッション／イベントのマスタデータは **JSON** で管理（後から教員が数値・文言を直しやすいスキーマにする）
- 当日はインターネット非依存で運用できるよう、**教員PC上でサーバーを起動し、タブレットは同一LAN内のブラウザからアクセス**する構成を基本とする（学校Wi-Fiが使えない場合に備え、モバイルルーター等の代替手段も準備）
- 進行状態はサーバー側のメモリ＋簡易なファイル/JSON永続化で保持し、途中で1台のタブレットが再接続してもゲーム状態を復元できるようにする

## 10. 確認したいこと（実装開始前にご回答いただきたい事項）

以下は方式検討のため`AskUserQuestion`で回答いただき、**確定済み**：
- タイブレーク：効率性→安全性の順（5章に反映）
- 場のリセット：毎ラウンドリセット（5章に反映）
- 運用形態：各グループがタブレットで入力（6〜8章に反映）

引き続き以下をご確認・ご提供ください。

1. **カードデータの完成度**：現状いただいているのは発電所カード4種のみ。残り16種、およびミッション／イベントカードの全ラインナップ（枚数・条件・効果文）はありますか？揃い次第JSON化します。揃うまでは4種のサンプルでプロトタイプを進めます。
2. **ラウンド数**：1ミッションで勝敗を決めて終わりか、複数ミッションを連続してポイントを競う形式か。
3. **グループ数・端末**：当日のグループ数（3〜4想定とのことでしたが確定数）、各グループのタブレットは学校/家庭のどの端末を使うか（台数・OS・ブラウザ）。
4. **ネットワーク環境**：教室で使える学校Wi-Fiがあるか、それとも教員PCのモバイルルーター等を持ち込む想定か（当日の接続方式に直結するため）。
